  1단계: 각 클래스가 "왜" 이렇게 설계됐는지 말로 설명할 수 있어야 해요

  모르는 게 있으면 그 부분을 직접 공부하는 게 1단계예요.

  ---
  2단계: 직접 실행하고 손으로 테스트해보기

  - Postman이나 curl로 업로드 → 실제 파일이 D:/images 어디에 생기는지 직접 확인
  - 잘못된 파일 올려보고 에러 응답 확인
  - 테스트 코드 실행하고 통과하는지 확인

  ---
  3단계: 직접 코드를 수정해보기 (가장 중요)

  코드를 읽는 것만으로는 내 것이 안 돼요. 직접 뭔가 고쳐봐야 해요. 예를 들어:

  - ALLOWED_EXTENSIONS에 .svg 추가해보기
  - project 파라미터가 빈 문자열이면 400 반환하는 validation 추가
  - MinioStorageService 스텁 코드를 보고 실제로 어떻게 구현해야 할지 생각해보기

  ---
  4단계: 회사 환경에 맞게 변경해야 할 부분 파악

  이 코드는 지금 범용 템플릿이에요. 실제 적용 전에 확인할 것들:

  - image.dir, image.url-prefix → 회사 서버 경로로 변경
  - Nginx와 X-Accel-Redirect 연동을 실제로 구성할 건지
  - 인증/인가가 필요한지 (지금 API에 보안이 전혀 없어요)
  - 로그 경로, 보관 기간 (logback-spring.xml)이 회사 기준에 맞는지

  ---
  결론

  코드를 분석해서 내 것으로 만들어야 하냐? → 네, 맞아요.

  다만 "처음부터 다시 짜야 한다"는 뜻은 아니에요. AI가 만들어준 코드를 설명할 수 있고, 수정할 수 있고, 문제가
  생겼을 때 디버깅할 수 있는 상태가 되는 게 목표예요. 그 전까지는 회사 프로덕션에 올리면 안 됩니다.

  위 질문들에 다 답할 수 있으면 그때 배포해도 늦지 않아요.

  "직접 수정해보기"의 핵심은 궁금한 게 생겼을 때 코드로 검증하는 습관이에요.

  관심 없는 부분까지 억지로 다 실험할 필요는 없어요. 오히려 역효과예요.

    다만 한 가지만 주의하면 돼요:

    실험 전에 "이렇게 하면 어떻게 될 것 같다"는 예측을 먼저 해보고, 결과가 예측과 다르면 왜 다른지 찾아보는 것.

    그냥 바꿔보고 "아 이렇게 되네" 하고 넘어가면 손만 바쁜 거고, 예측 → 실험 → 검증 사이클이 돌아야 진짜 이해한
    거거든요.

    "project 파라미터에 ../ 같은 경로 traversal 공격을 넣으면?" → 실제로 요청 날려보기 (보안 취약점 여부 확인
      겸)


    지금 하신 것과 매핑하면:

    흥미로운 점은 AI가 4~6단계를 대신 해줬다는 거예요.

    덕분에 속도는 엄청나게 빨라졌지만, 7~9단계는 AI가 대신해줄 수 없어요. 실제로 코드를 돌려보고, 막히고, 다시
    찾아보는 과정은 본인이 직접 해야 이해가 생기니까요.

    불안감의 정체가 바로 이거예요. 4~6단계를 건너뛴 것 같은 느낌. 근데 건너뛴 게 아니라 AI와 함께 처리한 거고,
    지금 7단계부터 제대로 밟고 있는 거예요.
